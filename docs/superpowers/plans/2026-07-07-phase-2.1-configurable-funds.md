# Phase 2.1 — Configurable Funds Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Funds become data, not code — a `funds` table drives tiles, pickers, the ledger engine, and auto-cover priority; snapshots move to per-fund `snapshot_balances` rows; a Funds management UI lands in the Plan panel; Vitest is bootstrapped with engine + migration tests (spec §3, §4, §13, §14 sub-phase 2.1).

**Architecture:** Extract the ledger engine from `server/index.js` into a pure module (`server/engine.js`) that takes rows in and returns the ledger — this is what makes it testable and fund-generic (balances as a `{key: amount}` map instead of five scalars). A one-shot idempotent migration (`server/migrate.js`) runs at server start: backs up the db, creates `funds` + `snapshot_balances`, seeds the four existing funds with today's colors and auto-cover order, and explodes legacy snapshot fund columns into balance rows. Frontend swaps every hardcoded fund array for data from `GET /api/funds`.

**Tech Stack:** sql.js (existing), Express, React 18, Vitest (new devDependency — the one §13 allows; node_modules is vendored in this repo, so the install is committed).

## Global Constraints

- Branch `feature/2.1-configurable-funds` off `feature/phase-1-ux-hotfixes`; PR #2 targets the phase-1 branch (stacked; GitHub retargets to main when PR #1 merges).
- **Stop the dev server before running/testing the migration** — the running sql.js process holds the whole db in memory and rewrites the file on any write, clobbering a migration done underneath it.
- Cash is not a fund (spec §4): it stays the special operating balance; fund keys are validated against a reserved list (`cash`, `openCash`, `events`, `dayIn`, `daySpend`, `dayTransfer`, `dayOut`, `hasSnapshot`, `cash_pinned`, `reconciled`, `autocoverOwed`) because day objects spread fund balances by key.
- Legacy snapshot columns (`car`,`emergency`,`debt`,`home`,`partial_funds`) stay physically in the table but are never read or written after migration (sql.js DROP COLUMN support is not guaranteed; spec's "slims to" is honored logically).
- Ledger API response keeps its shape: day objects still expose one balance per fund key at the top level, so Phase 1 frontend code keeps working during the transition.
- Migration must be idempotent: it runs only when the `funds` table is absent; a second server start is a no-op.
- Deviation from spec §4 noted: reorder uses ↑/↓ controls and auto-cover priority is a numeric field ("Never" = NULL) instead of drag-and-drop — no drag library exists in the repo and none may be added; drag can layer on later without API changes.
- Tests: `npm test` = `vitest run`. Engine and migration tests are required (spec §13); UI is browser-verified like Phase 1.
- (Author = executor in the same session with all files in context; steps below carry full contracts and core code, with UI edits specified by exact component/prop rather than full JSX restating.)

### Task 2: Migration — funds + snapshot_balances

**Files:**
- Create: `server/migrate.js`
- Modify: `server/index.js` (run migration at startup, after legacy schema/seed)
- Test: `server/migrate.test.js`

**Interfaces:**
- Produces: `migrate(db, { dbPath, backupsDir })` → `{ ran: boolean, backupPath?: string }`. Runs iff `funds` table absent.

Core contract:

```sql
CREATE TABLE funds (
  id TEXT PRIMARY KEY, key TEXT UNIQUE NOT NULL, label TEXT NOT NULL,
  color TEXT NOT NULL DEFAULT '#1D4ED8',
  sort_order INTEGER NOT NULL DEFAULT 0,
  autocover_priority INTEGER,          -- NULL = never auto-drawn
  target REAL,                         -- NULL = no goal line
  archived INTEGER NOT NULL DEFAULT 0
);
CREATE TABLE snapshot_balances (
  snapshot_id TEXT NOT NULL, fund_id TEXT NOT NULL,
  amount REAL NOT NULL DEFAULT 0,
  PRIMARY KEY (snapshot_id, fund_id)
);
ALTER TABLE snapshots ADD COLUMN cash_set INTEGER DEFAULT 0;
```

Steps inside `migrate()`:
1. If `funds` exists in `sqlite_master` → return `{ran:false}`.
2. Copy db file to `data/backups/finance-<YYYYMMDD-HHmmss>.db` (create dir).
3. Create tables + `cash_set` column.
4. Seed funds (colors/labels from the current `FUNDS` array, auto-cover order from the current engine): `('f_car','car','Car fund','#1D4ED8',0,4)`, `('f_emergency','emergency','Emergency','#047857',1,1)`, `('f_debt','debt','Debt Fund','#1E40AF',2,2)`, `('f_home','home','Tuition reserve','#6D28D9',3,3)`.
5. For every snapshot row: `partial = partial_funds.split(',')`, `full = partial.length === 0`; insert a `snapshot_balances` row per fund where `full || partial.includes(key)` with the legacy column amount; set `cash_set = full || partial.includes('cash') ? 1 : 0`.

- [ ] Step 1: Write `migrate.js` per contract; wire into `index.js` right after the demo seed (fresh dbs seed legacy-shape then migrate — one code path).
- [ ] Step 2: `migrate.test.js`: build an in-memory legacy db (schema from `index.js` + one full snapshot + one partial `partial_funds='car'` + one cash-partial), run migrate, assert: 4 funds seeded with priorities (emergency 1, debt 2, home 3, car 4); full snapshot → 4 balance rows + `cash_set=1`; partial → 1 row + `cash_set=0`; cash-partial → 0 rows + `cash_set=1`; second `migrate()` → `{ran:false}`, row counts unchanged. (In-memory db: pass `dbPath:null` → skip file backup.)

### Task 3: Engine goes dynamic

**Files:**
- Modify: `server/engine.js`
- Test: `server/engine.test.js` (new cases)

**Interfaces:**
- Consumes: `funds` rows (all, incl. archived) and `snapshotBalances` rows.
- Produces: day objects `{ cash, [each fund key]: number, ... }` — same shape as today for the four seeded funds.

Semantics:
- `bal = {}` keyed by fund key (archived funds included — history math unaffected, spec §4).
- Snapshot application: build `balancesBySnapshot` map; on a snapshot day, set `cash` iff `cash_set||cash_pinned`, set `bal[key]` for each balance row. **Full** = `cash_set` && a row for every non-archived fund → resets `autocoverOwed` (spec §3).
- Base seeding at `loopFrom`: per fund, nearest balance row at-or-before the base snapshot date (fixes the latent zero-fill when the base snapshot was partial); cash from nearest `cash_set` snapshot at-or-before.
- Auto-cover: draw from funds with `autocover_priority != null && !archived`, ascending priority (ties → `sort_order`); repayment exact reverse. `autocoverOwed` becomes a dynamic map.
- `debtpay` deducts from `bal['debt']` when a `debt`-keyed fund exists (archived or not), else from cash.
- `fund`/`source_fund` events resolve by key against `bal`.

- [ ] Step 1: Rewrite engine internals per semantics; `/api/ledger` now passes real `funds` + `snapshotBalances` rows.
- [ ] Step 2: New tests: custom priority order (car=1 drains first); NULL priority never drawn; archived fund never drawn; repay reverse of custom order; partial-snapshot anchoring (only listed fund moves); full snapshot resets owed; base-seeding pulls fund value from an older snapshot when base is partial. Run `npx vitest run` → green (characterization tests from Task 1 must still pass unchanged — that's the no-regression proof).
- [ ] Step 3: Restart server; calendar numbers identical to pre-migration. Commit: `feat: ledger engine driven by funds table (dynamic auto-cover, snapshot_balances)`.

### Task 4: Funds CRUD + snapshots API rework

**Files:**
- Modify: `server/index.js`
- Modify: `src/api.js`

**Interfaces (produces):**

```
GET    /api/funds                → [{id,key,label,color,sort_order,autocover_priority,target,archived,referenced}]
POST   /api/funds                {label,color,target?,autocover_priority?} → {id,key}   (key slugged from label, unique, reserved-list rejected 400)
PUT    /api/funds/:id            {label,color,target?,autocover_priority?,archived?}
POST   /api/funds/reorder        {ids:[...]} → sort_order = array index
DELETE /api/funds/:id            409 if referenced (rules/transactions fund_target|source_fund, snapshot_balances)
POST   /api/snapshots            {date, cash?, cash_pinned?, reconciled?, balances?:{key:amount}}
```

`referenced` = EXISTS over `rules.fund_target|source_fund`, `transactions.fund_target|source_fund`, `snapshot_balances.fund_id`. Snapshots handler: upsert row for date; `cash` present → set cash + `cash_set=1`; upsert a balance row per `balances` key (key→fund_id lookup); `partial_funds` never written. api.js adds `getFunds/addFund/updateFund/deleteFund/reorderFunds` and `upsertSnapshot` keeps its name with the new payload.
